# AV2技能可行性矩阵

## 1. 审计范围

本文件为`02_SKILL_LIBRARY_DESIGN_GOAL.md`的阶段B证据，判断候选技能能否由当前Argoverse 2 Motion Forecasting数据观察、作为真实种子、通过轨迹世界模型生成并自动评价。

审计仅使用：

- 课程题目5原文；
- AV2 Motion Forecasting数据结构和静态地图；
- `manifests/development/development_train.csv`中的500个Train开发场景；
- 仓库已核验的AV2、Trajectron++、AdvSim、KING和STRIVE资料；
- 交通规则语义和TTC、PET、时距、最小间距等安全代理指标。

未使用内部验证或最终Validation选择技能和阈值，也未扫描20,000个正式训练场景。

## 3. 当前可用与待补能力

### 3.1 原始数据能够提供

- 参与者ID、类别、位置、速度、航向和有效时间掩码；
- 5秒历史与6秒未来轨迹；
- 焦点参与者；
- 车辆、行人、骑行者、摩托车手、公交车等动态类别；
- 车道中心线、边界、前驱、后继、左右邻接、车道类型和路口属性；
- 人行横道和可行驶区域；
- coarse-grained的`static`、`background`和`construction`类别。

### 3.2 当前适配器尚未完整暴露

`skilldrive/data/av2_reader.py`目前只转换车道边界和中心线，后续03阶段若经确认技能需要，还应最小化补充：

- 人行横道几何；
- 可行驶区域边界；
- 车道前驱、后继和左右邻接ID；
- 车道标线类型；
- 必要时保留原始地图对象引用或结构化元数据。

这些字段原始JSON已经存在，因此属于工程缺口，不是数据缺失。

### 3.3 当前数据无法可靠提供

- 摄像头图像、视频和遮挡可见性；
- 雨、雾、眩光、低照度等外观条件；
- 相机损坏、模糊、曝光或传感器退化；
- 动态交通灯相位和精确信号状态；
- 临时限速、临时封道和施工组织的可靠语义；
- `static`或`construction`对象的精细类别、轮廓和物理尺寸；
- 车辆是否为警车、救护车等应急车辆。

## 4. 可行性等级

- A：可从AV2轨迹与地图直接观察，可筛选自然或边界种子，可由轨迹生成并自动评价。
- B：AV2可提供兼容基础场景，但长尾行为主要依靠反事实生成；仍可通过轨迹与地图自动评价。
- C：缺少决定性观测或评价字段，当前轨迹路线无法可靠实现。

A类与B类候选合计必须不少于30类。C类可以记录，但不计入30类。

## 6. C类排除项

以下项目可以保留在范围说明中，但不计入30类：

| 排除项 | 等级 | 原因 |
|---|:---:|---|
| 大雨、浓雾、眩光、低照度 | C | 无摄像头或天气外观数据，轨迹无法证明环境条件 |
| 相机模糊、遮挡、损坏、曝光异常 | C | 无传感器图像及质量状态 |
| 道路遗落物和低矮障碍物精细类别 | C | static类别缺少精细语义、轮廓和尺寸 |
| 锥桶导流、临时封道、临时改道、临时限速 | C | 静态地图不提供可靠动态施工组织语义 |
| 闯红灯 | C | 缺少完整动态交通灯相位和信号状态 |
| 应急车辆优先权 | C | 参与者类别不能可靠识别警车、救护车等身份 |
| “遮挡后显现”的视觉真实性 | C | 可以生成路侧进入或前车切出，但不能验证真实可见性遮挡 |

其中“道路阻塞”和“施工影响”仍可通过B类的`construction_object_lane_blockage`与`static_object_avoidance`表达，但必须使用保守名称，不声称当前数据无法证明的精细对象类别。

## 7. 结论与待确认事项

- 当前数据足以支持以轨迹交互、车道拓扑、路口、人行横道、可行驶区域和弱势参与者为核心的30类以上技能体系。
- 40个A/B候选均不依赖摄像头外观或动态信号灯；20个A类用于自然/边界样本筛选，20个B类主要用于条件反事实生成。
- `uncontrolled_crossing_conflict`的“无控制”无法由动态信号状态证明，建议最终改名为更保守的`crossing_path_conflict`，或者明确仅表示几何无显式优先信息。
- construction/static类别可支持阻塞与绕行，但不能细分为锥桶、碎片或低矮障碍物。
- 摩托车手、部分自行车和多参与者组合候选可能种子稀缺，应作为用户确认时的重点取舍项。
