# SkillDrive 完整实施计划

## 1. 项目目标与成功标准

本项目采用 **Argoverse 2 Motion Forecasting + 条件 CVAE 轨迹世界模型 + BEV 场景表达 + 轻量矢量 Transformer 预测验证**。

课程题目允许从视频、BEV/占用网格或轨迹世界模型中选择一种生成路线，并要求至少选择一个下游任务验证，因此本项目不生成摄像头视频，而是生成带有地图、轨迹、风险和技能标签的结构化长尾驾驶场景。

最终需要完成：

- 建立不少于30类完整正式技能规则；当前经确认的正式规则库包含34类，其中A类14个、B类20个，另保留5类本轮零命中的候选规则。正式与候选合计39个已实现规则。C类只能作为排除说明且不计数，不能以名称占位代替实现。
- 39个已实现规则共享少量可复用的检测、轨迹编辑和质量验证算子，但每类都必须具有独立的触发条件、控制参数、约束、风险指标和期望行为。
- 从正式Train原始候选池中确定性筛选5,000个唯一场景种子，并保留所选场景的全部正式技能标签。
- 为每类正式技能生成300–1,000个变体，形成约10,000–50,000条长尾轨迹样本；实际批量规模按资源分阶段执行，但不得静默缩减34类正式技能。5类候选规则在满足晋级条件前不进入正式训练或批量生成。
- 人工复核500–1,000条高质量样本，并尽量保证全部34类正式技能有代表性覆盖；其中至少100条由两人重复审核。5类候选规则只有在后续获得合格种子并经用户确认晋级后，才进入生成与评估。
- 使用真实数据、随机增强、未过滤技能增强、过滤后技能增强进行公平对比。
- 在总体测试集和真实长尾子集上分别报告性能，不能只展示 BEV 动画或总体平均指标。
- 目标是使过滤后的技能数据让长尾子集上的 minFDE 或 Miss Rate 相对改善 5% 以上，同时总体性能下降不超过 2%。如果未达到目标，必须保留完整负结果、失败归因和数据有效性分析。

课题要求来源：`大数据综合课程实践课题-大数据-2026.docx` 中的题目 5。

## 2. 系统设计

### 2.1 数据与场景表示

- 只使用 Argoverse 2 Motion Forecasting 数据，不下载摄像头和激光雷达数据。
- 使用 5 秒历史轨迹和 6 秒未来轨迹，采样频率为 10 Hz。
- 所有场景转换到目标参与者最后一个历史观测时刻的局部坐标系。
- 模型输入包括最多 32 个邻近参与者、100 米范围内的车道折线、参与者类别、位置、速度、航向和有效性掩码。
- 训练前按照 `scenario_id` 固定划分数据，验证场景不得作为生成种子。
- 从官方Train split确定性下载22,000个场景，其中20,000个用于真实训练、2,000个用于内部验证，两者按`scenario_id`互斥。
- 从20,000个正式训练场景和2,000个内部验证场景中分别固定派生500/100开发子集，不重复下载或复制数据。
- 从官方Validation split确定性下载5,000个场景作为最终测试子集；39规则扫描中命中的34类已经全部进入正式规则库，实现34/34类正式技能种子覆盖。其余5类零命中规则保留为候选，因此不再为当前正式规则覆盖扩大下载规模。
- 采用分片处理：按照场景清单读取原始数据，处理成紧凑的矢量分片，不同时长期保存全部原始缓存、BEV 图片和生成中间结果。

### 2.2 技能规则库

每个技能使用统一的 YAML 配置格式：

```yaml
skill_id:
name_zh:
family:
definition:
source:
data_support:
seed_requirements:
trigger:
actors:
parameters:
generation_operators:
constraints:
risk_definition:
expected_behavior:
validation_metrics:
known_limitations:
output_labels:
```

技能的具体名单由`02_SKILL_LIBRARY_DESIGN_GOAL.md`完成候选方案、来源说明和AV2可行性矩阵，并在用户确认后固化；当前版本为34类正式技能与5类候选规则。技能选择遵循以下原则：

- 题目示例、交通规则、安全指标、相关研究和Train数据模式共同提供候选来源；
- 技能语义和规则结构由人工定义，数值阈值优先使用Train统计校准；
- 每类技能必须能够由当前轨迹世界模型表示、生成并自动评价；
- AV2 Motion Forecasting无法观察或验证的纯视觉、天气和传感器退化技能不得仅以占位形式计入正式规则数；
- 34类正式技能与5类候选规则可以共享纵向运动、换道、停车、汇入、冲突点穿越、让行、弱势参与者运动和局部通行约束等基础算子；
- 共享算子不等于合并技能，每个技能仍需保留独立条件、参数、风险语义和评价方式。

当前v5条件CVAE和05阶段正式合同每次只生成一个关键参与者的未来轨迹，其他背景参与者保持真实，从而形成开放环局部反事实编辑。双目标联合生成只有在单目标阶段完成后另立模型升级合同并重新训练，不能通过拼接两条独立轨迹实现。

### 2.3 条件 CVAE 轨迹世界模型

默认模型结构：

- 参与者历史编码：GRU，隐藏维度 128。
- 地图折线编码：逐点 MLP 加池化，输出维度 128。
- 场景交互编码：2 层、4 头轻量 Transformer。
- 技能 ID 嵌入：32 维。
- 连续技能参数接口：MLP 编码为32维；当前v5正式训练的`parameter_mask`全部为0，因此该接口尚未形成已验证的连续参数控制能力。
- 潜变量：16 维高斯变量。
- 解码器：当前v5 GRU生成单个目标参与者的60步位置增量；未来双目标版本需要独立结构、监督和checkpoint。
- 损失函数：当前v5正式训练启用Smooth L1轨迹重建、KL退火和终点约束；地图与碰撞软损失保留实现接口，但正式权重为0，不能据此宣称模型已经学会地图或碰撞约束。
- 推理时当前v5通过不同潜变量和有真实监督证据的技能ID生成候选；未受监督的连续参数只用于规则搜索目标和生成后实现度检查，不得称为模型直接参数控制。

规则只负责提供条件、可控参数和约束，未来轨迹必须由学习模型生成，避免退化为简单的坐标平移式数据增强。

### 2.4 质量过滤

以下情况直接拒绝：

- 出现 NaN、缺帧、参与者身份数量异常或轨迹不连续；
- 速度、加速度、jerk、曲率或转向率超过按参与者类别、技能语义和真实训练分布冻结的物理硬边界；99.5%分位只作为初始校准证据之一，不机械套用全部技能；
- 车辆明显驶出适用道路空间，或违反该技能不允许违反的车道方向；逆行等目标技能必须使用技能感知例外但仍保持在可行驶道路范围；
- 出现非预期碰撞；
- 技能触发条件未满足；
- 风险过低，没有形成实际长尾挑战。

软质量评分由以下部分组成：

- 运动学一致性；
- 地图一致性；
- 交互真实性；
- 技能控制成功率；
- 风险等级匹配；
- 与同一种子其他样本的多样性。

软质量分用于同类候选排序和平衡选择；是否设置接受分数线及其数值必须在05阶段Pilot中依据冻结指标校准，不能预先把全局0.75作为所有技能的共同真值。高风险场景以近失碰撞为主，不主动生成不可避免的实际碰撞；TTC 1–4秒只适用于目标风险定义确实使用TTC的技能，其余技能按各自YAML风险指标和范围评价。

### 2.5 BEV 输出

BEV 不作为像素生成模型，而作为统一的场景表达、质量审核和结果展示接口。每个典型案例需要展示：

- 车道中心线、道路边界和路口；
- 参与者历史轨迹；
- 真实未来轨迹；
- 条件 CVAE 生成轨迹；
- 风险发起者和受影响者；
- 技能 ID、技能参数、TTC、过滤结果和期望行为。

批量 BEV 结果保存在 Git 忽略目录中；每类选择少量代表案例放入 `assets/`，用于 README 和课程报告。

### 2.6 下游轨迹预测模型

正式下游模型采用轻量矢量 Transformer：

- GRU 编码参与者历史；
- MLP 加池化编码地图折线；
- 2 层交互 Transformer；
- 输出 6 条候选未来轨迹及其概率；
- 使用最佳轨迹回归损失和模式分类损失。

同时保留以下基础模型：

- 恒速模型：验证坐标转换和指标实现是否正确。
- LSTM 预测器：快速验证数据增强管线。
- QCNet 等公开强基线只作为时间充裕时的加分项，不阻塞主线。

主要命令行接口：

```text
prepare_data
build_manifests
detect_seeds
train_generator
generate_scenarios
filter_scenarios
render_bev
train_predictor
evaluate_predictor
run_ablation
```

所有命令必须读取 YAML 配置，禁止硬编码本地路径。生成结果必须记录配置哈希、Git 提交、随机种子和源场景 ID。

## 3. 实验与评价

### 3.1 正式对比组

- E0：仅使用真实数据；
- E1：真实数据加等数量随机扰动；
- E2：真实数据加未过滤的技能生成数据；
- E3：真实数据加过滤后的技能生成数据。

随机增强只进行无语义的坐标噪声、速度缩放和合法范围内的横向偏移，样本数量必须与 E2、E3 一致。

### 3.2 消融实验

- 去掉技能 ID；
- 去掉连续技能参数；
- 将潜变量固定为零，取消生成多样性；
- 比较 2,500、5,000 和 10,000 条生成数据；
- 分析每类正式技能的独立增益；
- 逐类移除正式技能；
- 关闭质量过滤。

### 3.3 评价指标

轨迹预测使用 6 条候选轨迹，报告：

- minADE；
- minFDE；
- 6 秒终点、2 米阈值的 Miss Rate；
- 总体测试集、各技能对应的真实长尾切片及不同风险等级的分组结果。

生成质量报告：

- 生成接受率；
- 技能触发成功率；
- 碰撞率和地图违规率；
- 速度、加速度和轨迹连续性；
- 样本间平均轨迹距离；
- TTC 风险区间覆盖；
- 人工真实性、可控性、多样性和任务有效性评分。

人工审核由团队成员完成，其中至少 100 条由两人重复审核，以检查评分一致性。

## 4. 实施阶段与分工

### 第 1 周：环境和数据闭环

- 使用 WSL2 Ubuntu 24.04、`uv`管理的Python 3.10；GPU版PyTorch `2.13.0+cu130`已经作为项目正式依赖锁定。
- 建立环境锁定文件、配置系统和测试框架。
- 下载22,000个Train场景，划分20,000个正式训练和2,000个内部验证场景，并派生500/100开发清单。
- 完成单个场景的读取、局部坐标转换和 BEV 绘制。
- 验收：同一场景经过全局坐标到局部坐标再回到全局坐标后，误差小于 `1e-4`。

### 第 2 周：不少于30类技能体系设计（当前34正式+5候选）

- 从题目示例、交通规则、安全指标、相关研究和Train数据模式提出不少于30类候选技能。
- 为每类建立来源、现实含义、AV2可观察性、轨迹可生成性和自动可评价性说明。
- 设计共享检测与生成算子，避免为每类重复开发独立模型。
- 向用户提交完整候选清单和取舍依据，经确认后才固化技能YAML。
- 验收：经用户确认的34类正式技能与5类候选规则均具有完整规则模板、数据支持说明和实现路径，不存在名称占位；正式与候选状态明确且互斥。

### 第 3 周：规则执行和种子检测

- 实现经确认技能所需的共享触发器、风险计算和参数采样器。
- 先扫描500个开发训练场景，输出可解释触发依据并完成分层BEV审核。
- 用户确认规则方向后扫描20,000个正式训练场景，先保留完整原始候选池，再建立最终种子清单。
- 验收：获得恰好5,000个唯一候选种子并保留全部正式技能标签；若候选不足则形成有证据的停止说明；种子清单不包含5类候选规则、内部验证场景或最终Validation场景。

### 第 4 周：条件 CVAE 基线（已完成）

- 完成场景编码、潜变量、技能条件和局部轨迹解码。
- 先在单技能和小型开发集上进行过拟合验证，并完成500/100开发训练闭环。
- 使用固定负载持续优化预处理和稳定训练吞吐，冻结当前硬件下最快且正确的配置后，完成20,000个正式训练场景和2,000个内部验证场景的可续训全流程训练。
- 验收：模型能在小数据上明显降低重建损失，针对不同潜变量生成不同未来轨迹；训练性能优化具有可复核基准，正式训练和内部验证完整结束。
- 当前结果：v5正式缓存、50轮训练、独立内部验证和16例诊断已经完成，全量pytest、源码编译、`git diff --check`和一致性审计均已通过，本阶段已关闭。

### 第 5 周：批量生成与过滤（正式数值阶段已完成，审核交付待完成）

- 初始v5 checkpoint固定为单目标解码器；阶段D可以在保持单目标50/60步外部schema的前提下形成经过重新验收的active checkpoint。第一版只替换一个明确目标参与者的未来，其他参与者未来保持真实背景，不将其宣传为闭环或联合生成。
- 实现Prior候选生成、轻量轨迹overlay、运动学、地图、碰撞、风险、技能有效性、参数实现度和多样性过滤。
- 区分有真实监督的直接条件Prior与无真实监督技能的规则引导Prior搜索，不把未训练Embedding宣传为已经学会控制。
- 先对34类正式技能执行分层Pilot、条件对照和BEV审核；能力不足时先优化规则引导搜索与推理约束，必要时修正训练合同并训练新版本checkpoint，随后重新完成正确性验收。
- 正确性冻结后再用固定负载反复优化GPU生成、CPU过滤和端到端吞吐；综合绝对节时、相对变化和重复波动确认已无明显收益后冻结配置，随后直接进入正式批量任务。
- 正式任务按冻结预算处理5,000个种子和33,914条技能标签，每类最多平衡选择300条合格变体；无法达到数量目标时保留完整负结果，不降低质量门槛。
- 验收：生成、过滤、元数据、性能证据和断点续传已经由正式合同验证；代表性可视化和人工审核仍在阶段G执行。
- 进入条件：04阶段最终审计通过，初始v5数据合同、最佳checkpoint、生成随机种子和过滤口径冻结；阶段D验收后再冻结正式使用的active checkpoint与active contract。Final Validation在本阶段继续禁用，不得读取、调参或筛样。

### 第 6 周：下游预测基线

- 完成恒速、LSTM 和轻量矢量 Transformer。
- 在纯真实数据上训练并验证指标。
- 验收：训练损失正常收敛，预测结果优于恒速基线，官方验证场景不存在数据泄漏。

### 第 7 周：正式对比实验（待阶段G/H完成后启动）

- 按已确认规模生成10,000–50,000条技能数据，并构造等规模随机增强和未过滤数据；资源受限时先完成10,000条主实验，再分批扩展。
- 运行 E0–E3 四组实验。
- 固定模型、训练轮数、随机种子和真实数据量，只改变增强数据。
- 验收：结果表、日志、模型权重和配置一一对应。

### 第 8 周：消融与人工审核

- 完成条件、过滤、规模和技能类别消融。
- 完成500–1,000条人工复核并覆盖全部34类正式技能，其中至少100条由两人重复审核。
- 输出典型成功和失败案例。
- 验收：可以解释哪些技能有效、哪些过滤器关键、哪些样本导致负迁移。

### 第 9 周：报告与复现检查

- 从空环境重新运行一个小型端到端流程。
- 整理结果 CSV、图表、BEV 案例和失败归因。
- 更新 README、复现命令和报告素材。
- 验收：仅修改数据根目录即可完成预处理、生成、过滤、训练和评估。

### 团队分工

2–3 人团队默认分工：

- 成员 A：AV2 数据、坐标系、地图、BEV 和种子检测；
- 成员 B：技能规则、条件 CVAE 和质量过滤；
- 成员 C：预测模型、实验管理、指标和结果可视化；
- 如果只有两人，成员 A 兼任技能触发器，成员 B 兼任预测和评估。

## 5. 测试、仓库与交付

- 39份规则配置的字段完整性、来源和数据支持检查，以及34类正式技能与5类候选规则的状态分区检查；
- 共享触发与生成算子的正例、反例和边界测试；
- 每类技能至少一个代表性规则映射测试；
- 技能参数采样边界；
- CVAE 输出形状、有限值和确定性随机种子；
- 地图、碰撞、运动学和 TTC 过滤器；
- 训练和验证场景 ID 零交集；

- 单场景端到端集成测试；
- 小批量训练和评估冒烟测试。

- 可运行代码及环境文件；
- 经确认的34类正式技能配置、5类候选规则配置及其共享规则算子实现；

- 轻量结果 CSV 和选定的 BEV 案例；

## 6. 默认假设与范围边界

- 不额外等待教师确认，按照题目中的“或”和“建议”理解执行。
- 第一版不生成摄像头视频，不训练栅格 BEV 或占用网格世界模型。
- 第一版只验证轨迹预测，占用网格、规划和闭环仿真属于加分项。
- 主要训练使用本机 RTX 4060 Laptop 8GB，通过混合精度、小批量、梯度累积和分片数据控制资源。
- 项目运行环境为 WSL2 Ubuntu 24.04、`uv`管理的Python 3.10；GPU依赖按训练阶段实际驱动安装。
- 不以保证正向结果为前提；如果增强无效，必须保留严谨实验，并分析生成分布偏差、过滤缺陷和负迁移原因。

## 7. Goal 模式执行顺序

Goal 模式应严格按以下顺序推进：

1. 检查当前仓库和本计划，不重复已完成工作。
2. 建立环境、目录、配置和最小测试框架。
3. 首先跑通一个 AV2 场景的端到端数据闭环。
4. 先提出A类与B类合计不少于30类的候选技能（本轮实际提出40类）、完成AV2可行性论证并获得用户确认，不预先把自行选择的技能写成既定方案。
5. 对经确认的技能实现共享规则算子和候选种子检测，再开发条件CVAE。
6. 在小型开发集通过验收后，先优化并冻结训练性能配置，再完成20,000/2,000正式生成器训练；正式模型验收后才启动批量生成。
7. 对生成和过滤使用固定负载持续优化，冻结综合时间与稳定性最优配置后完成正式批量任务；完成质量过滤后，再训练正式下游预测模型。
8. 固定实验配置后运行对比和消融，禁止根据结果临时改变测试集。
9. 每完成一个阶段，更新README、测试和本文件中的进度记录。
10. 未达到阶段验收标准时不得直接跳到下一阶段。
