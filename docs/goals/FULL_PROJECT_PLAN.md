# SkillDrive 完整实施计划

## 1. 项目目标与成功标准

本项目采用 **Argoverse 2 Motion Forecasting + 条件 CVAE 轨迹世界模型 + BEV 场景表达 + 轻量矢量 Transformer 预测验证**。

课程题目允许从视频、BEV/占用网格或轨迹世界模型中选择一种生成路线，并要求至少选择一个下游任务验证，因此本项目不生成摄像头视频，而是生成带有地图、轨迹、风险和技能标签的结构化长尾驾驶场景。

最终需要完成：

- 建立包含 30 类场景定义的技能目录，其中 5 类具备完整可执行生成能力。
- 从真实数据中筛选不少于 2,000 个候选场景种子。
- 生成并过滤约 10,000 条长尾轨迹样本，每类核心技能约 2,000 条。
- 人工复核 500 条高质量样本，每类 100 条。
- 使用真实数据、随机增强、未过滤技能增强、过滤后技能增强进行公平对比。
- 在总体测试集和真实长尾子集上分别报告性能，不能只展示 BEV 动画或总体平均指标。
- 目标是使过滤后的技能数据让长尾子集上的 minFDE 或 Miss Rate 相对改善 5% 以上，同时总体性能下降不超过 2%。如果未达到目标，必须保留完整负结果、失败归因和数据有效性分析。

课题要求来源：`大数据综合课程实践课题-大数据-2026.docx` 中的题目 5。

## 2. 系统设计

### 2.1 数据与场景表示

- 只使用 Argoverse 2 Motion Forecasting 数据，不下载摄像头和激光雷达数据。
- 使用 5 秒历史轨迹和 6 秒未来轨迹，采样频率为 10 Hz。
- 所有场景转换到目标参与者最后一个历史观测时刻的局部坐标系。
- 模型输入包括最多 32 个邻近参与者、100 米范围内的车道折线、参与者类别、位置、速度、航向和有效性掩码。
- 训练前按照 `scenario_id` 固定划分数据，验证场景不得作为生成种子。
- 从官方Train split确定性下载22,000个场景，其中20,000个用于真实训练、2,000个用于内部验证，两者按`scenario_id`互斥。
- 从20,000个正式训练场景和2,000个内部验证场景中分别固定派生500/100开发子集，不重复下载或复制数据。
- 从官方Validation split确定性下载5,000个场景作为最终测试子集；如果五类技能的候选种子不足，再以5,000个Train场景为一批增量下载。
- 采用分片处理：按照场景清单读取原始数据，处理成紧凑的矢量分片，不同时长期保存全部原始缓存、BEV 图片和生成中间结果。

### 2.2 技能规则库

每个技能使用统一的 YAML 配置格式：

```yaml
skill_id:
family:
implemented:
trigger:
actors:
parameters:
constraints:
risk_definition:
expected_behavior:
output_labels:
```

30 类技能目录分为六组，每组五类：

- 车辆交互：切入、急减速、跟停、汇入、无保护转向冲突。
- 弱势参与者：行人横穿、骑行者横穿、逆行骑行、路侧突然进入、群体横穿。
- 道路障碍：临停车辆、遗落物、低矮障碍、车道变窄、车道阻塞。
- 施工变化：施工绕行、临时封道、锥桶导流、临时改道、临时限速。
- 环境传感器：降雨、浓雾、眩光、低照度、相机退化。
- 规则优先权：未让行、闯灯、应急车辆、四向停车歧义、无控制路口抢行。

第一版完整实现以下五类技能，视觉类技能只保留目录定义：

1. 相邻车辆切入；
2. 前车急减速；
3. 慢车或停车阻塞；
4. 行人或骑行者横穿；
5. 汇入或路口让行。

每次只联合生成 1–2 个关键参与者的未来轨迹，其他背景参与者保持真实，从而形成局部反事实编辑。

### 2.3 条件 CVAE 轨迹世界模型

默认模型结构：

- 参与者历史编码：GRU，隐藏维度 128。
- 地图折线编码：逐点 MLP 加池化，输出维度 128。
- 场景交互编码：2 层、4 头轻量 Transformer。
- 技能 ID 嵌入：32 维。
- 连续技能参数：MLP 编码为 32 维。
- 潜变量：16 维高斯变量。
- 解码器：GRU 生成 60 步位置增量，最多输出两个关键参与者。
- 损失函数：Smooth L1 轨迹重建、KL 退火、终点约束、地图软约束和碰撞软约束。
- 推理时通过不同潜变量和技能参数，为同一场景生成多个反事实变体。

规则只负责提供条件、可控参数和约束，未来轨迹必须由学习模型生成，避免退化为简单的坐标平移式数据增强。

### 2.4 质量过滤

以下情况直接拒绝：

- 出现 NaN、缺帧、参与者身份数量异常或轨迹不连续；
- 速度、加速度或转向率超过对应参与者类别在真实训练集中的 99.5% 分位范围；
- 车辆明显驶出可行驶区域或违反车道方向；
- 出现非预期碰撞；
- 技能触发条件未满足；
- 风险过低，没有形成实际长尾挑战。

软质量评分由以下部分组成：

- 运动学一致性；
- 地图一致性；
- 交互真实性；
- 技能控制成功率；
- 风险等级匹配；
- 与同一种子其他样本的多样性。

只有不存在硬性错误且总分不低于 0.75 的样本，才能进入过滤后数据集。高风险场景以近失碰撞为主，默认 TTC 目标范围为 1–4 秒，不主动生成不可避免的实际碰撞。

### 2.5 BEV 输出

BEV 不作为像素生成模型，而作为统一的场景表达、质量审核和结果展示接口。每个典型案例需要展示：

- 车道中心线、道路边界和路口；
- 参与者历史轨迹；
- 真实未来轨迹；
- 条件 CVAE 生成轨迹；
- 风险发起者和受影响者；
- 技能 ID、技能参数、TTC、过滤结果和期望行为。

批量 BEV 结果保存在 Git 忽略目录中；每类选择少量代表案例放入 `assets/`，用于 README 和课程报告。

### 2.6 下游轨迹预测模型

正式下游模型采用轻量矢量 Transformer：

- GRU 编码参与者历史；
- MLP 加池化编码地图折线；
- 2 层交互 Transformer；
- 输出 6 条候选未来轨迹及其概率；
- 使用最佳轨迹回归损失和模式分类损失。

同时保留以下基础模型：

- 恒速模型：验证坐标转换和指标实现是否正确。
- LSTM 预测器：快速验证数据增强管线。
- QCNet 等公开强基线只作为时间充裕时的加分项，不阻塞主线。

主要命令行接口：

```text
prepare_data
build_manifests
detect_seeds
train_generator
generate_scenarios
filter_scenarios
render_bev
train_predictor
evaluate_predictor
run_ablation
```

所有命令必须读取 YAML 配置，禁止硬编码本地路径。生成结果必须记录配置哈希、Git 提交、随机种子和源场景 ID。

## 3. 实验与评价

### 3.1 正式对比组

- E0：仅使用真实数据；
- E1：真实数据加等数量随机扰动；
- E2：真实数据加未过滤的技能生成数据；
- E3：真实数据加过滤后的技能生成数据。

随机增强只进行无语义的坐标噪声、速度缩放和合法范围内的横向偏移，样本数量必须与 E2、E3 一致。

### 3.2 消融实验

- 去掉技能 ID；
- 去掉连续技能参数；
- 将潜变量固定为零，取消生成多样性；
- 比较 2,500、5,000 和 10,000 条生成数据；
- 分析每类技能的独立增益；
- 逐类移除技能；
- 关闭质量过滤。

### 3.3 评价指标

轨迹预测使用 6 条候选轨迹，报告：

- minADE；
- minFDE；
- 6 秒终点、2 米阈值的 Miss Rate；
- 总体测试集、各技能对应的真实长尾切片及不同风险等级的分组结果。

生成质量报告：

- 生成接受率；
- 技能触发成功率；
- 碰撞率和地图违规率；
- 速度、加速度和轨迹连续性；
- 样本间平均轨迹距离；
- TTC 风险区间覆盖；
- 人工真实性、可控性、多样性和任务有效性评分。

人工审核由团队成员完成，其中至少 100 条由两人重复审核，以检查评分一致性。

## 4. 实施阶段与分工

### 第 1 周：环境和数据闭环

- 使用 WSL2 Ubuntu 24.04、`uv`管理的Python 3.10；训练前再安装兼容的GPU版PyTorch。
- 建立环境锁定文件、配置系统和测试框架。
- 下载22,000个Train场景，划分20,000个正式训练和2,000个内部验证场景，并派生500/100开发清单。
- 完成单个场景的读取、局部坐标转换和 BEV 绘制。
- 验收：同一场景经过全局坐标到局部坐标再回到全局坐标后，误差小于 `1e-4`。

### 第 2 周：技能目录和种子检测

- 建立 30 类技能配置。
- 完成 5 类核心技能的触发器和参数采样器。
- 扫描正式训练清单，建立每类候选种子清单。
- 验收：每类输出可解释的触发依据，并人工检查至少 20 个种子。

### 第 3 周：条件 CVAE 基线

- 完成场景编码、潜变量、技能条件和局部轨迹解码。
- 先在单技能和小型开发集上进行过拟合验证。
- 验收：模型能在小数据上明显降低重建损失，并针对不同潜变量生成不同未来轨迹。

### 第 4 周：批量生成与过滤

- 完成 1–2 个关键参与者的反事实生成。
- 实现运动学、地图、碰撞、风险和技能有效性过滤。
- 生成首批每类 200 条合格数据并完成 BEV 审核。
- 验收：生成、过滤、元数据和可视化可以由一个配置命令完整复现。

### 第 5 周：下游预测基线

- 完成恒速、LSTM 和轻量矢量 Transformer。
- 在纯真实数据上训练并验证指标。
- 验收：训练损失正常收敛，预测结果优于恒速基线，官方验证场景不存在数据泄漏。

### 第 6 周：正式对比实验

- 生成约 10,000 条过滤后数据及等规模随机增强和未过滤数据。
- 运行 E0–E3 四组实验。
- 固定模型、训练轮数、随机种子和真实数据量，只改变增强数据。
- 验收：结果表、日志、模型权重和配置一一对应。

### 第 7 周：消融与人工审核

- 完成条件、过滤、规模和技能类别消融。
- 完成 500 条人工复核。
- 输出典型成功和失败案例。
- 验收：可以解释哪些技能有效、哪些过滤器关键、哪些样本导致负迁移。

### 第 8 周：报告与复现检查

- 从空环境重新运行一个小型端到端流程。
- 整理结果 CSV、图表、BEV 案例和失败归因。
- 更新 README、复现命令和报告素材。
- 验收：仅修改数据根目录即可完成预处理、生成、过滤、训练和评估。

### 团队分工

2–3 人团队默认分工：

- 成员 A：AV2 数据、坐标系、地图、BEV 和种子检测；
- 成员 B：技能规则、条件 CVAE 和质量过滤；
- 成员 C：预测模型、实验管理、指标和结果可视化；
- 如果只有两人，成员 A 兼任技能触发器，成员 B 兼任预测和评估。

## 5. 测试、仓库与交付

### 5.1 自动测试

必须包含：

- 坐标变换往返一致性；
- 变长参与者和地图掩码；
- 五类技能触发的正例和反例；
- 技能参数采样边界；
- CVAE 输出形状、有限值和确定性随机种子；
- 地图、碰撞、运动学和 TTC 过滤器；
- 训练和验证场景 ID 零交集；
- 相同配置和随机种子生成结果一致；
- 单场景端到端集成测试；
- 小批量训练和评估冒烟测试。

### 5.2 仓库交付内容

- 可运行代码及环境文件；
- 30 类技能配置和 5 类完整实现；
- 数据清单、字段定义和生成配置；
- 轻量结果 CSV 和选定的 BEV 案例；
- 质量审核记录；
- 完整 README 和复现命令。

不得提交：

- 原始 AV2 数据；
- 完整模型权重；
- 大规模生成结果；
- 本地绝对路径；
- 密钥和个人配置。

## 6. 默认假设与范围边界

- 不额外等待教师确认，按照题目中的“或”和“建议”理解执行。
- 第一版不生成摄像头视频，不训练栅格 BEV 或占用网格世界模型。
- 第一版只验证轨迹预测，占用网格、规划和闭环仿真属于加分项。
- 主要训练使用本机 RTX 4060 Laptop 8GB，通过混合精度、小批量、梯度累积和分片数据控制资源。
- 项目运行环境为 WSL2 Ubuntu 24.04、`uv`管理的Python 3.10；GPU依赖按训练阶段实际驱动安装。
- 不以保证正向结果为前提；如果增强无效，必须保留严谨实验，并分析生成分布偏差、过滤缺陷和负迁移原因。

## 7. Goal 模式执行顺序

Goal 模式应严格按以下顺序推进：

1. 检查当前仓库和本计划，不重复已完成工作。
2. 建立环境、目录、配置和最小测试框架。
3. 首先跑通一个 AV2 场景的端到端数据闭环。
4. 实现技能目录及种子检测，再开发条件 CVAE。
5. 在小型开发集通过验收后，才启动正式批量生成。
6. 完成质量过滤后，再训练正式下游预测模型。
7. 固定实验配置后运行对比和消融，禁止根据结果临时改变测试集。
8. 每完成一个阶段，更新 README、测试和本文件中的进度记录。
9. 未达到阶段验收标准时不得直接跳到下一阶段。
