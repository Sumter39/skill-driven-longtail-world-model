# SkillDrive 不少于30类技能体系设计 Goal（当前34正式+5候选）

## 1. 当前目标

本阶段依据课程题目、交通安全知识、相关研究和Argoverse 2 Motion Forecasting的数据能力，提出A类与B类合计不少于30类的长尾驾驶技能候选方案，完成来源说明、数据可行性论证、共享实现方式设计，并在用户确认后固化为完整技能规则库。可以额外列出C类作为排除项，但C类不计入30类数量。

本阶段解决“30类技能具体是什么、为什么选择、AV2是否支持、以后如何生成和评价”的问题，不进行正式候选种子扫描，不训练世界模型，也不批量生成长尾轨迹。

## 2. 核心原则

- “不少于30类”来自课程题目，具体分类由项目组设计，不存在教师给出的固定30项名单。
- 技能语义和规则结构以人工定义为主，Train数据统计用于验证可行性和校准数值阈值。
- 不允许仅用名称占位来满足30类数量要求。
- 每类技能都必须能够由当前轨迹世界模型表示、条件生成并自动评价。
- AV2无法观察或验证的纯视觉、天气和传感器退化技能不能直接计入正式规则库，除非后续明确扩展数据与模型范围。
- 30类技能可以共享少量规则与生成算子，不需要30个独立模型。
- 在用户确认候选清单前，不把任何具体30类名单写成最终方案。
- A类与B类候选合计必须不少于30类；C类只用于记录排除原因，不计入候选数量或最终技能数量。

## 3. 规则来源

每个候选技能必须至少标明一种来源：

- `course_example`：课程题目明确举例；
- `traffic_rule`：交通法规、道路优先权或规范驾驶行为；
- `safety_metric`：TTC、PET、急减速、最小间距等安全指标；
- `literature`：长尾驾驶、轨迹预测、反事实生成等相关研究；
- `train_pattern`：AV2 Train数据中可验证的运动或交互模式。

仅有直觉名称、没有现实定义和证据来源的候选技能不得进入最终清单。

## 4. AV2可行性审查

每个候选技能必须回答：

1. AV2中是否能观察到所需参与者、轨迹和地图关系？
2. 是否能找到真实场景作为种子，或找到兼容的基础场景进行反事实生成？
3. 条件CVAE能否通过关键参与者轨迹表达该技能？
4. 是否能用轨迹、地图和风险指标自动验证生成结果？
5. 是否依赖AV2不提供的摄像头外观、天气、传感器状态或动态道路设施？

候选技能按可行性分级：

- A：能够从AV2直接观察、筛选、生成和评价；
- B：需要从兼容基础场景进行反事实生成，但仍可由轨迹和地图评价；
- C：当前数据或轨迹表示无法可靠表达。

正式规则库只允许选择A类和论证充分的B类，数量不得少于30；当前定稿为34类正式技能与5类零命中候选规则。C类可以单独列出用于解释范围边界，但不能以`implemented: false`占位计数。

## 5. 候选方案输出

在创建最终技能YAML前，必须先向用户提交A类与B类合计不少于30类的候选技能方案。可以另附C类排除清单，但不得用C类补足30类数量。每个候选至少包含：

```text
候选编号
技能英文ID
技能中文名称
技能族
现实场景定义
规则来源
主要参与者
所需AV2字段
可行性等级
真实种子要求
共享检测或生成算子
主要控制参数
风险指标
期望驾驶行为
已知限制
与相近技能的区别
```

候选方案必须同时给出：

- 分类体系是否覆盖题目举出的主要风险方向；
- 哪些题目示例因轨迹数据限制无法直接采用；
- 哪些候选容易语义重复；
- 哪些候选可能缺少真实种子；
- 推荐保留、合并或替换的理由。

这是强制交流点。用户确认、删除、合并或替换候选技能后，才能继续固化最终规则库。

## 6. 共享实现方式

候选技能需要映射到少量可复用的基础能力，例如：

- 纵向加速、减速和停车；
- 横向换道和切入；
- 车道汇入与分流；
- 冲突点穿越；
- 让行、抢行和优先权变化；
- 弱势参与者横穿或进入道路；
- 局部车道占用和通行约束；
- 多参与者时序协调。

共享算子只负责复用几何和生成机制。每个技能仍必须拥有不同的触发语义、参与者角色、参数范围、风险目标、约束和期望行为。

## 7. 数值阈值原则

规则中的数值必须注明来源：

- `semantic`：由交通安全含义直接定义，例如风险目标区间；
- `train_statistics`：根据Train数据分布、分位数或聚类结果确定；
- `reference`：来自标准、论文或公开评测定义。

允许使用500个开发训练场景进行轻量统计和字段覆盖检查，但不得在本阶段正式扫描20,000个场景，也不得使用内部验证或最终Validation调节规则。

## 8. 最终技能配置

用户确认最终清单后，正式技能与保留候选规则均建立完整YAML，至少包含：

```yaml

```

现有`configs/skills/catalog.yaml`和5份示例YAML只作为前期结构验证产物，不自动视为最终技能选择。是否保留其中任何技能，由本阶段候选审查决定。

## 9. 输出文件

本阶段预计形成：

```text
docs/skills/skill-candidates.md
docs/skills/av2-feasibility-matrix.md
docs/skills/skill-taxonomy.md
configs/skills/catalog.yaml
configs/skills/candidate_catalog.yaml
configs/skills/<正式技能或候选规则>.yaml
tests/unit/skills/test_skills.py
```

`skill-candidates.md`记录提交给用户的候选方案和取舍过程；`skill-taxonomy.md`记录用户确认后的34类正式技能、5类候选规则及二者边界。

## 10. 执行步骤

### 阶段A：需求与现状审计

- 对照课程题目整理技能数量、种子规模、生成规模和质量要求；
- 审查现有30类目录和5份YAML，区分题目要求、前期假设和数据不支持项；
- 汇总AV2可用参与者类别、轨迹字段和地图能力。

### 阶段B：提出候选方案

- 从五类规则来源提出A类与B类合计不少于30类的候选技能，并另行记录必要的C类排除项；
- 完成可行性等级、技能边界和共享算子映射；
- 检查语义重复、不可观察条件和实现风险；
- 输出完整候选表，不直接改成最终技能配置。

### 阶段C：用户确认

- 向用户说明候选分类依据和取舍；
- 对存在分歧的技能提供保留、合并、替换选项；
- 等待用户确认最终不少于30类清单；
- 未确认前不得进入阶段D。

### 阶段D：固化规则库

- 为确认后的正式技能与保留候选规则建立完整YAML；
- 建立最终分类说明、正式/候选状态分区和AV2可行性矩阵；
- 标注阈值来源、共享算子、限制和后续验证方式；
- 删除不再采用的占位定义，但不得删除仍需保留的历史说明。

### 阶段E：测试与交付

- 验证技能ID唯一、字段完整、参数范围合法；
- 验证每类均有来源、数据支持和实现路径；
- 验证没有C类技能被计入最终数量；
- 运行全部自动测试和`git diff --check`；
- 更新README和阶段状态。

## 11. 验收标准

只有同时满足以下条件，本Goal才能完成：

- 已向用户提交A类与B类合计不少于30类的完整候选方案；C类如有列出，未计入该数量；
- 用户已确认最终不少于30类正式技能清单，并明确保留候选规则的范围；
- 每类正式技能与候选规则均有明确来源、现实定义和与相近技能的区别；
- 每类正式技能与候选规则均通过AV2可观察性、轨迹可生成性和自动可评价性论证；
- 所有正式技能均为A类或论证充分的B类，候选规则不计入正式数量；
- 每类均映射到共享实现能力，但保留独立规则语义；
- 每类均具有完整YAML，不存在仅名称占位；
- 数值阈值均注明来源；
- 自动测试和文档检查通过；
- 未正式扫描20,000个场景，未训练模型，未批量生成轨迹。

## 12. Goal模式规则

1. 首先读取课程题目、本文件、`01_PREPARATION_GOAL.md`和`FULL_PROJECT_PLAN.md`。
2. 不把现有目录或前期5份YAML当作已经确认的最终方案。
3. 先输出候选方案和证据，再请求用户确认。
4. 遇到技能语义、数据支持或分类边界分歧时提供选项，不静默决定。
5. 用户确认前不得批量创建最终技能配置。
6. 不使用内部验证或最终Validation选择技能和阈值。
7. 不训练模型，不正式扫描20,000个场景，不扩大数据下载规模。
8. 未经用户明确授权，不提交或推送Git。
